# 审计界面卡片与时间线布局优化设计

## 背景
当前 `webapp/src/views/AdminAuditView.vue` 已支持跨会话轮次查看审计回放，但审计页仍有几个明显问题：
- 轮次选择使用横向按钮条，占用较多横向与纵向空间；
- 时间线条目中文映射不完整，仍会暴露 `interaction.requested`、`run.waiting` 等原始标签；
- “对话信息”和“执行信息”分成两个顶部卡片，高度占用偏大；
- 时间线卡片与明细卡片的留白较多，右侧明细区域对 JSON / payload 展示仍然偏紧。

用户希望这一轮在不打破现有 chat / audit 视觉语言的前提下，提升审计页紧凑度与信息可读性，尤其为时间线明细区域释放更多展示空间。

## 目标
1. 将轮次选择改为与 chat 页面模型选择一致的下拉菜单交互。
2. 为审计事件补充完整中文展示名称，优先通过后端返回可展示操作名，前端保留回退能力。
3. 将顶部“对话信息”和“执行信息”合并为一个摘要卡，并支持折叠。
4. 让主内容区更偏向“明细优先”：左侧时间线更窄，右侧明细更宽。
5. 全程复用现有设计语言，避免原生控件和不匹配的圆角、边框、阴影、配色。

## 非目标
- 不重做审计页筛选逻辑；
- 不引入新的通用 UI 库；
- 不改变审计事件的底层语义结构；
- 不为摘要折叠态增加“记住上次状态”的持久化。

## 现状观察
- 轮次选择位于 `webapp/src/views/AdminAuditView.vue` 的 `admin-audit-turn-bar`，当前是横向按钮组。
- chat 页在 `webapp/src/views/ChatView.vue` 已有成熟的 `model-menu` 交互，可直接借用结构和样式。
- 审计页中文文案当前主要依赖 `formatEventType()` 与 `formatPhase()` 本地映射；映射不全时会直接展示原始值。
- 后端回放接口 `app/handlers/audit_handler.go` 当前返回的 `AuditReplayBundle` / `AuditReplayEvent` 只有原始事件字段，没有前端专用展示名。
- 现有样式集中在 `webapp/src/style.css`，其中已有一套 admin audit 与 model-menu 风格可复用。

## 方案概述

### 1. 顶部摘要卡：合并 + 默认折叠
将“对话信息”和“执行信息”合并成单个摘要卡，放在主内容区顶部。

折叠态：
- 默认进入页面时折叠；
- 仅展示少量高价值字段：创建者、轮次数、Run ID、状态；
- 用一行紧凑布局呈现，减少垂直空间占用。

展开态：
- 展示原先两张卡的完整字段；
- 仍使用 definition list 语义，但布局压缩为更紧凑的多列信息块；
- 保持与现有卡片体系一致的圆角、边框和背景。

交互：
- 使用一个轻量的折叠按钮切换展开 / 收起；
- 不做持久化记忆，本次仅保留页面内状态。

### 2. 轮次选择：按钮条改下拉菜单
将现有 `admin-audit-turn-bar` 按钮条替换为下拉菜单。

行为：
- 仅在 `auditRuns.length > 1` 时显示；
- 默认选项为“全部轮次”；
- 其余选项为“轮次 1 / 轮次 2 / ……”；
- 选中某个轮次后，维持现有 `selectTurn()` 的过滤逻辑。

样式策略：
- 复用 `ChatView.vue` 的 `model-menu` 结构、交互语义和视觉样式；
- 避免使用浏览器原生 `select` 样式；
- 审计页只做最小定制，保证与 chat 页统一。

### 3. 中文操作名称：后端优先，前端回退
为避免审计页持续手工维护事件名映射，采用“后端提供展示名，前端优先显示”的方案。

后端：
- 在审计回放时间线事件结构上增加 `display_name` 展示字段，语义固定为“给 UI 显示的中文操作名称”；
- 后端继续保留原始 `event_type`、`phase`、`payload`，确保可排查性；
- 为常见事件补全中文展示名，至少覆盖当前已知缺失项，如 `interaction.requested`、`run.waiting` 等。

前端：
- 时间线主标题优先显示后端返回的展示字段；
- 副信息保留原始 `event_type`，方便审计和排查；
- 若后端字段缺失，则回退到前端 `formatEventType()` 映射；
- 若本地映射也未命中，最终再回退到原始 `event_type`。

这样可以兼顾：
- 新事件扩展时前端更稳定；
- 页面可读性提升；
- 原始技术信息仍然可见。

### 4. 时间线与明细：明细优先布局
主内容区继续保留左右两栏，但调整为“明细优先”：
- 左侧时间线列收窄；
- 右侧时间线明细列加宽；
- 时间线头部整合为一行紧凑控制区：标题、轮次下拉、筛选按钮并列。

时间线条目排版：
- 第一行：中文操作名称；
- 第二行：原始 `event_type` + `phase` + `seq` + `轮次`；
- 保留现有 tone / icon 体系；
- 缩小无效 padding 和 gap，提高单屏可见条目数。

时间线明细排版：
- 标题区保留动态标题；
- 元信息行压缩高度；
- JSON / payload / artifact 正文区域尽量吃满剩余空间；
- 保持现有 `pre` 风格基础上做紧凑化，避免额外视觉分叉。

### 5. 视觉一致性要求
本次改动必须遵守以下约束：
- 优先复用现有 `webapp/src/style.css` 中的 card、menu、chip、button 风格；
- 不使用浏览器原生默认控件外观；
- 不引入与 chat / admin audit 不匹配的圆角、边框、阴影和新配色；
- 若新增审计页专用样式，需以现有颜色体系与半透明卡片体系为基础做局部调整。

## 数据流
1. 前端加载会话与对应 runs；
2. 前端请求 replay bundle；
3. 后端在构建 replay timeline 时为每个事件附带展示名；
4. 前端时间线渲染时：
   - 主标题：`display_name ?? formatEventType(event_type) ?? event_type`
   - 副标题：保留 `event_type / phase / seq / turnIndex`
5. 用户切换轮次下拉或过滤器时，仅改变现有前端计算属性筛选结果。

## 错误处理
- 如果后端暂时未返回展示名，前端必须正常显示，不得出现空标题；
- 若轮次只有一轮，不显示轮次下拉；
- 若摘要所需字段缺失，保留当前页面的 `-` / fallback 文案策略；
- 若时间线为空，继续展示现有 empty state 文案。
